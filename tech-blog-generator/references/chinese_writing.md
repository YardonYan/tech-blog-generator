# Chinese Technical Writing Rules (中文技术写作规则)

Adapted from luoling8192/technical-writing. These rules apply when generating Chinese technical content.

## Core Principles

### 1. 先观察，再归纳，再判断 (Observe → Induct → Judge)
不要抢在读者前面下结论。写作三步：
- 观察到了什么 (what we observed)
- 可以归纳出什么 (what can be inducted)
- 这些归纳落在哪里 (where these inductions land)

### 2. 信息承载量检验
遇到任何句子，先问：删掉它，论证会不会受影响？如果不会 — 直接删掉，不要换成语。

### 3. 证据先行
如果要给出较强判断，必须满足至少一项：
- 前文已经提出对应猜想，这一段在 callback
- 同一段上下文已有交叉验证的 evidence
- 已引用具体模块、代码、issue、trace、指标或实验结果

## Structure: 长文主线

优先用这条主线组织长文：
```
背景 → 拆解现状 → 提出猜想 → 分节解答 → 回指前文 → 影响面 → 可观测性/eval → 收束
```

如果文章围绕系统展开，每一节都应写成「在回答前面哪个问题」，让读者知道当前段落在解决什么。

## TL;DR 三段式 (写给 Leader/评审)

如果需要在最开头放 TL;DR，用三段式：
1. 先讲启发来源或背景变化
2. 再讲当前系统已具备哪些输入、还缺哪一层结构
3. 最后讲新模块设计和架构决策

TL;DR 只保留三类信息：**原因、猜想、决策**。不解释接下来会怎么写。

## Sentence-Level Rules

### 中文标点规范
- 中文引号统一用「」
- 不使用破折号 `—` 和英文连字符 `-` 做分句
- 专有概念按正式名字书写，大小写稳定：Agent Runtime、Context Engine、Agent Signal

### 避免的句式 (Sentence Patterns to Avoid)

#### 主持口吻 (Host Tone)
这些句子不承载信息，直接删除，不改写：
- ❌ 「聊到这里，memory 相关模型也该放进来一起看了」
- ❌ 「先把『压缩』这件事单独拿出来说」
- ❌ 「这张图想说明的事情很简单」
- ❌ 「接下来进入这一节真正关心的问题」
- ❌ 「最后可以用一句更轻一点的话结束」

处理：直接删除，让下一句作为段落开头。

#### 抢结论的词 (Premature Judgment Words)
- ❌ 很清楚、说明了、显然、真正、自然会
- ✅ 更稳妥的写法：「顺着这个观察，可以归纳出...」「前面提到的猜想，在这里开始有了更具体的回答」

#### 对立句法 (Oppositional Syntax)
- ❌ 「不是……而是……」「不在……而在……」
- 为什么避免：容易写出说教感。读者还没进入正文就被带节奏。
- 改法：去掉对立结构，保留重音。（例：「更值得认真拆开的，是它把一组运行时机制组织得比较完整」）

#### 空评价词 (Empty Evaluations)
- ❌ 很现实、很重、很轻、很顺
- 这些只有情绪，没有信息。改法：先写条件，再写对象，最后写判断。
  - 例：「可以预见的是，Agent Signal 集成进入 adapter 之后，信号控制会很重要。」

#### 过场句 (Transitional Fluff)
- ❌ 「聊到这里」「继续往下看」「这条方向很值得继续推开」「先把这件事单独拿出来说」「最后可以用一句话收住」
- 处理：如果一句话只是在主持文章、不承载信息，直接删掉。

## Buzzword Blocklist (黑话词表)

看到这些词先停一下，能换成具体动作/对象/约束就换掉。不是逐词替换，而是逐句重写。

| 黑话 | 替代方向 |
|------|---------|
| 生态/生态位/产品生态 | 列出具体依赖关系、用户类型或集成方 |
| 赋能/赋能用户 | 让用户能 X / 把 X 能力开放给 Y |
| 闭环/形成闭环 | X 之后 Y 也接进去了 / 现在能从 A 走到 B 再回到 A |
| 抓手/核心抓手 | 我们能改的那一个变量是 X |
| 心智/用户心智 | 用户在第一次看到 X 时会怎么想/怎么操作 |
| 链路打通/打通 | A 现在能调用 B / A 写入的数据 B 也能读到 |
| 沉淀/沉淀经验 | 记录到 docs/X / 整理成 skill/runbook |
| 落地/落地能力 | 在 A 模块实现 / 在 B 业务用上 |
| 心力/认知负担 | 需要记住 X 个步骤 / 上手要先理解 Y |
| 收口/统一收口 | 所有 X 都走 Y 一个入口 |
| 加持/AI 加持 | 加了 X 之后能 Y / 在 X 步用了 LLM |
| 底座/底层逻辑 | 基础设施层是 X / 这件事成立的前提是 Y |
| 很值得/非常值得 | 给具体理由：值得在哪个维度上 |
| 极致/极致体验 | 改成具体指标：延迟降到多少、步骤减到几步 |
| 丝滑/顺畅 | 改成具体约束：哪步原本卡，现在跳过了 |
| 深度集成/深度对接 | 改成集成点的具体范围 |
| 一站式 | 在 A 入口能完成 B/C/D |
| 颠覆/重塑 | 默认不用；真要用得有 before/after 对比 |

## Formatting Rules

### 适合列出来的内容
不要为了「像文章」而把 list-shaped 内容硬写成整段 prose：
- 设计目标
- 猜想
- 模块影响面
- 待补充实现点
- 评测命题
- 适用/不适用场景

### 论证顺序
如果一组判断有论证顺序，优先写成 `1. 2. 3. 4.`，不要散落成并列 prose。

### 适合用 quote 的内容
- 想强调的命题
- 读者可能先想到的另一种问题定义
- 关键句摘录

### 适合用箭头的内容
- `source → signal → action`
- `background → hypothesis → answer`
- `observe → infer → route → write back`

## 抽象层与实现层

只谈抽象容易飘，只谈代码会失去设计视角。技术文档通常需要两层一起出现：
- **抽象层**：概念、边界、语义、主张
- **实现层**：模块、类型、executor、policy、storage、trace、eval

写实现层时，把模块面摊开：
- 哪些模块已经存在
- 哪些模块只体现出局部规律
- 哪些模块更适合作为后续替代实现
- 哪些地方需要补偿、迁移、压缩、解耦

## 语气 (面向内部评审/Leader/前线开发)

- 少下定论，多给出推演路径
- 少用绝对化副词，多用「先」「可以」「暂时」「顺着这个观察」
- 少写审判式句子，多写解释式句子
- 少写「作者赢了」的结尾，多写「读者可以继续往下跟」的结尾

## 交稿前检查清单 (Pre-submission Checklist)

- [ ] 有没有没有证据支撑的强判断
- [ ] 有没有「不是……而是……」一类对立句法
- [ ] 有没有「很清楚/说明了/显然/真正」这种抢结论的词
- [ ] 有没有把代码直接翻译成抽象结论
- [ ] 有没有只讲概念，不讲模块和影响面
- [ ] 有没有该列清单的地方却硬写成长段
- [ ] 有没有前文提出的猜想，后文却没有 callback
- [ ] 指标名、动作词、状态词有没有混用
- [ ] 有没有过场句在替内容占位置——删掉试试论证会不会受影响
- [ ] 中文引号有没有统一成「」
